NexCard Project Overview
Generated: 2026-06-23

1. Project Summary

NexCard is a digital smartcard / digital visiting card application. It lets a user create an account, manage a public profile from an admin dashboard, customize the design of the card, and publish the card at a public username URL such as /username.

The public card can show profile details, contact actions, social links, services, gallery images, videos, custom links, payment details, QR codes, map location, and an enquiry form. The card supports multiple visual themes.

The repository is organized as a monorepo:

- frontend: Next.js app with React, Tailwind CSS, Firebase Auth, Firestore, and Firebase Storage initialization.
- backend: Express API placeholder server.

2. Tech Stack

Route: /admin

Purpose:
- Authentication entry for admins/users.

Supported auth flows:
- Email/password sign in.
- Email/password signup.
- Google sign-in via popup, with redirect fallback.
- Password reset email.

Signup flow:
1. User enters email/password.
2. App derives a username from the email prefix.
3. It checks Firestore collection usernames/{username}.
4. If taken, it appends random digits.
5. Creates Firebase Auth user.
6. Writes username mapping to usernames/{cleanUsername}.
7. Writes basic user document to users/{uid}.
8. Redirects to /admin/dashboard.

Google login flow:
1. Attempts signInWithPopup.
2. If popup is blocked/cancelled/closed, falls back to signInWithRedirect.
3. processUserLogin derives username from email.
4. Ensures username mapping exists or creates suffixed username if needed.
5. Merges Google profile data into users/{uid}.
6. Adds default theme only if the user does not already have one.
7. Redirects to /admin/dashboard.

Password reset:
- Requires email field to be filled.
- Calls sendPasswordResetEmail(auth, email).

Route: /[username]

Purpose:
- Public card route.

Flow:
1. Uses Next 16 style use(params) to unwrap dynamic params in a client component.
2. Cleans username by lowercase, trim, and removing non-alphanumeric chars.
3. Fast lookup in Firestore usernames/{cleanUsername}.
4. If mapping missing, fallback query users where username equals cleanUsername.
5. Reads users/{uid}.
6. If username is the demo username and demoReady is missing, merges demo data locally.
7. Renders NexCard with profile data.

13. Known Risks and Improvement Areas

Security / privacy:
- ImgBB API key is hardcoded client-side in dashboard code. This should be moved to a backend/API route or replaced with Firebase Storage rules if uploads must be controlled.
- Firebase project config is client-side, which is normal for Firebase, but Firestore/Auth/Storage rules must be strict.
- Public card data is loaded directly from Firestore. Security rules should allow only intended public fields for public profiles.
- Dashboard writes the full form object to users/{uid}; validation is mostly client-side.
- Username reservation and deletion happen client-side and may need transaction/rules support to avoid race conditions.

Data integrity:
- Username cleanup removes all non-a-z0-9 characters. This is consistent across signup, dashboard, and public lookup.
- Signup checks username once and appends random digits if taken. There is still a possible collision if the suffixed username exists.
- Updating usernames deletes old mapping before/around writing new mapping. A safer approach would use a transaction or callable backend function.
- The public page has fallback query against users collection if username mapping is missing, which is useful but may be slower and requires an index/rules.

Architecture:
- Dashboard page is still large even after initial cleanup. It would benefit from further extraction:
  - ProfileTab component.
  - BusinessTab component.
  - SocialTab component.
  - MediaTab component.
  - ThemePresetsTab component.
  - DesignSettingsTab component.
  - UploadField component.
  - OptionGrid/OptionButton components.
- Theme components have repeated logic for actions, vCard generation, enquiry form, social icons, QR/payment rendering, gallery, videos, and maps. Shared utilities/components would reduce duplication.
- BottomNav has a lot of layout-specific styling logic in one component. A mapping object or per-layout style module would make it easier to maintain.

UX:
- Many user feedback actions use alert(). A toast system would feel smoother.
- The enquiry form opens mailto instead of storing/sending leads.
- Profile views are hardcoded in ModernTheme.
- Upload progress is simulated/manual rather than based on actual network progress.
- Public card uses img tags instead of Next Image, which may be fine for external URLs but loses optimization.

Compatibility:
- There is mojibake/encoding corruption in several comments and strings. Examples include emoji/comment text and copyright symbols. This should be cleaned to UTF-8 or plain ASCII.
- The project intentionally uses Next 16 APIs, including use(params) on the public dynamic route.

Backend:
- Backend currently does not provide useful API features.
- Potential backend responsibilities:
  - Secure uploads.
  - Lead capture.
  - username reservation transactions.
  - analytics/profile views.
  - payment webhook handling.
  - email notifications.

Testing:
- No explicit tests are present.
- Recommended tests:
  - Auth signup username generation.
  - Public username lookup.
  - Dashboard save payload.
  - Theme selection routing in NexCard.
  - vCard generation.
  - Share URL generation.
  - Basic render smoke tests for all themes.

18. Detailed Dashboard Flow

Initial load:
1. Dashboard component mounts.
2. loading is true.
3. onAuthStateChanged subscribes to Firebase Auth.
4. If no authenticated user:
   - router.push("/admin").
5. If authenticated user:
   - user state is set.
   - Firestore users/{uid} is fetched.
   - If no document exists, data defaults to empty object.
   - Demo profile logic may merge demo data.
   - setForm merges fetched data into INITIAL_FORM defaults.
   - loading is set to false.

Form merge behavior:
- Top-level profile fields from Firestore overwrite default form values.
- social merges prev.social with data.social.
- payment merges prev.payment with data.payment.
- theme merges prev.theme with data.theme.
- services/gallery/customLinks/videos default to empty arrays if missing.

Field updates:
- handleChange updates top-level fields by input name.
- handleNestedChange updates nested objects like social, payment, or theme.
- handleArrayChange replaces one array item by index.
- handleCustomLinkChange updates one key in one custom link object.
- addArrayItem appends an item to an array field.
- removeArrayItem filters an item out by index.

Upload flow:
1. User chooses an image file.
2. handleImageUpload reads e.target.files[0].
3. uploading true, progress set to 30.
4. File is posted to ImgBB upload endpoint.
5. progress set to 80 after response.
6. If success:
   - gallery index is updated, or
   - image is updated, or
   - coverImage is updated, or
   - payment.qrCode is updated.
7. Finally uploading false and progress 0.

Save flow:
1. User clicks Save Changes.
2. handleSave checks user exists.
3. Writes { ...form, uid: user.uid } to users/{uid}.
4. Shows alert on success or failure.

Demo fill flow:
1. User clicks Fill Demo Data.
2. confirm dialog is shown.
3. createDemoProfile called.
4. Current form is merged with demo data.
5. Current image is preserved if already present.
6. Current social/payment/theme values override demo values in the merge.

Username update flow:
1. User enters edit mode.
2. New username is cleaned.
3. If length < 3, alert.
4. usernames/{clean} is checked.
5. If taken by another user, alert.
6. Old username mapping is deleted if form.username exists.
7. New mapping is written.
8. User document is merged with new username.
9. Local form username updated.

Tab reset flow:
- Each tab reset asks for confirmation.
- themes reset only layout and color indexes.
- profile clears identity/photos.
- business clears contact/payment.
- social clears social links.
- media clears arrays.
- design merges DEFAULT_THEME.

19. Detailed Public Profile Flow

Incoming request:
- User visits /someusername.
- Next routes this to app/[username]/page.jsx.
- Component unwraps params with use(params).

Lookup:
1. cleanUsername = username.toLowerCase().trim().replace(/[^a-z0-9]/g, "").
2. Try direct document usernames/{cleanUsername}.
3. If found, read uid.
4. If not found, query users collection where username == cleanUsername.
5. If uid remains null, stop loading and show not found.
6. Read users/{uid}.
7. If found, set profile data.

Render:
- NexCard chooses theme.
- Theme renders full public card.
- BottomNav gives section navigation and share drawer.

Error states:
- Firestore error sets error to "Failed to connect to database", but the rendered not-found UI only checks !data. The error state is currently not prominently displayed.

24. Recommended Firestore Rule Intent

Not present in repo, but the app needs rules equivalent to:

- users/{uid}
  - Allow authenticated user to read/write their own document.
  - Allow public read of safe profile fields if public cards are intended.
  - Prevent users from writing uid that does not match request.auth.uid.

- usernames/{username}
  - Allow public read for lookup.
  - Allow create only if authenticated and document does not exist and uid == request.auth.uid.
  - Allow delete only if authenticated and existing uid == request.auth.uid.
  - Allow update rarely or never; prefer delete/create transaction or backend function.

Because the public page reads users/{uid} directly, either public reads are required on users documents or the architecture needs a publicProfiles collection with sanitized data.

31. Final Product Understanding

NexCard is currently a strong prototype/MVP for a digital business card platform. The frontend contains nearly all product logic. The main user journey is:

1. Visitor lands on marketing page.
2. Visitor creates account or logs in.
3. App creates a username and default profile.
4. User edits card details in dashboard.
5. User customizes theme/design.
6. User saves profile to Firestore.
7. User shares /username link or QR.
8. Public visitor views the card, saves contact, opens links, pays via UPI QR, navigates to location, or sends an enquiry.

The biggest next step is not adding more features immediately; it is making the existing feature set safer and more maintainable. The app already has a lot of surface area. Production readiness depends on stronger data rules, secure uploads, validation, better component boundaries, and a real backend or Firebase Functions layer for sensitive workflows.
